# Use Codacy for Quality Visibility and New-Change Gates

## Context and Problem Statement

The repository already has several focused CI gates: Spotless formats Java and
Kotlin source, Java compilation runs Error Prone checks, the Gradle test suite
guards functional regressions, Trivy scans repository secrets, Trivy scans the
built service image for fixed high and critical vulnerabilities, Kubernetes
manifest rendering validates the local deployment path, OpenAPI snapshot
generation detects API drift, and Prometheus rule checks validate local
observability artifacts.

The missing quality signal is not another compiler or security scanner. The
repository needs a lightweight way to see coverage trends, code smells,
maintainability, complexity, and duplication, especially for new pull request
changes.

Which repository quality visibility layer should the project use, and how
should it integrate with the existing CI gates?

## Decision Drivers

* The quality tool should add coverage visibility, trend tracking,
  maintainability, complexity, duplication, and code-smell signals.
* Existing narrow CI gates should remain authoritative for formatting,
  compilation, Error Prone analysis, tests, secret scanning, image
  vulnerability scanning, Kubernetes manifest rendering, OpenAPI drift, and
  Prometheus rule validation.
* The rollout should prioritize new pull request changes rather than forcing an
  immediate cleanup of all legacy findings.
* The service runs on Java 25, so JVM coverage tooling must support Java 25
  bytecode.
* The selected service should fit this open-source repository without requiring
  a paid rollout for the initial quality visibility layer.
* GitHub Actions dependencies should remain pinned by full commit SHA where
  applicable, and tool versions should be explicit.
* Coverage upload must not expose Codacy tokens broadly in the job environment.
* Pull request coverage upload must not use `pull_request_target` for workflows
  that check out or execute untrusted pull request code.
* Fork pull requests should still run normal CI when coverage upload secrets are
  unavailable.

## Considered Options

* Codacy with JaCoCo coverage upload and new-change-first gates
* SonarQube Cloud
* Existing CI only
* Replace existing CI checks with Codacy

## Decision Outcome

Chosen option: **Codacy with JaCoCo coverage upload and new-change-first
gates**, because Codacy fills the repository's remaining quality visibility gap
with an open-source plan that fits this open-source repository, without replacing
the existing focused CI checks.

Codacy is used for:

* coverage visibility and trend tracking;
* code smells and best-practice findings;
* maintainability, complexity, and duplication signals;
* pull request quality feedback;
* quality gates that initially focus on new changes.

Spotless, Error Prone, Gradle tests, Trivy, `k8sRenderDev`,
springdoc OpenAPI generation, and `promtool` continue to own formatting, Java
compiler analysis, functional tests, repository secret scanning, image
vulnerability scanning, Kubernetes manifest rendering, OpenAPI drift detection,
and Prometheus rule validation.

Coverage is generated in the `service` module with JaCoCo. The build pins the
JaCoCo tool version through the Gradle version catalog and configures
`jacocoTestReport` to produce XML and HTML reports. JaCoCo instruments JVM
bytecode, so the same report covers the service's Java and Kotlin classes.
JaCoCo 0.8.14 officially supports Java 25, and the repository pins the newer
0.8.15 release. The first Kotlin coverage baseline should still be treated as
calibration before adding exclusions or tightening thresholds, because
compiler-generated bytecode and inline behavior can affect reported coverage.

GitHub Actions uploads the JaCoCo XML report to Codacy after tests run. The
coverage upload action is pinned by full commit SHA, and the Codacy Coverage
Reporter version is pinned explicitly. The job exposes only a non-secret
availability flag, `CODACY_PROJECT_TOKEN_AVAILABLE`, at job scope. The actual
`CODACY_PROJECT_TOKEN` is passed only to the Codacy upload action input, and the
upload step is skipped when the token is unavailable. No `pull_request_target`
workflow is introduced for Codacy coverage upload.

Initial Codacy quality gates should prevent new debt before requiring broad
legacy cleanup. The initial rollout should gate new high-severity issues, new
security issues, and diff coverage after the Codacy signal is stable. Overall
coverage, overall grade, legacy issue count, complexity, and duplication should
start as observed signals and become hard gates only after reviewing the first
baseline. Codacy grade and coverage badge URLs are available and are included
in both `README.md` and `README.ja.md`; future badge changes should continue to
use stable URLs from Codacy.

### Consequences

* Good, because coverage, maintainability, complexity, duplication, and
  code-smell trends become visible without weakening existing CI gates.
* Good, because new-change-first gates prevent newly introduced debt while
  avoiding a disruptive legacy-code cleanup at rollout time.
* Good, because JaCoCo XML generation is local, reproducible, and independent of
  Codacy's cloud analysis lifecycle.
* Good, because coverage upload is guarded for missing secrets, so fork pull
  requests can still run normal CI.
* Good, because the upload action and downloaded reporter are both pinned,
  reducing supply-chain drift in CI.
* Bad, because Codacy quality gates depend on external service availability and
  repository integration state.
* Bad, because coverage for Kotlin bytecode can require baseline review before
  thresholds are meaningful.
* Bad, because Codacy status checks and badges require repository-side setup in
  Codacy before the full rollout is visible in GitHub.
* Neutral, because SCA, SBOM, license checks, DAST, AI policy checks, and
  container scanning remain out of scope for this Codacy rollout and can be
  decided separately.

### Confirmation

* `gradle/libs.versions.toml` pins `jacoco = "0.8.15"`.
* `service/build.gradle.kts` applies the `jacoco` plugin and sets
  `jacoco { toolVersion = libs.versions.jacoco.get() }`.
* `service/build.gradle.kts` configures `jacocoTestReport` to depend on
  `test`, emit XML and HTML reports, and skip CSV output.
* `.github/workflows/ci.yaml` runs
  `./gradlew :service:jacocoTestReport --no-daemon` after the compile and
  formatting checks.
* `.github/workflows/ci.yaml` uploads
  `service/build/reports/jacoco/test/jacocoTestReport.xml` with
  `codacy/codacy-coverage-reporter-action` pinned by full commit SHA.
* `.github/workflows/ci.yaml` pins `coverage-reporter-version: 14.1.3`.
* `.github/workflows/ci.yaml` guards the upload step with the non-secret
  `CODACY_PROJECT_TOKEN_AVAILABLE` flag and passes `secrets.CODACY_PROJECT_TOKEN`
  only as the Codacy action input.
* `.github/workflows/ci.yaml` keeps `contents: read` permissions and
  `persist-credentials: false` checkouts.
* `.github/workflows/ci.yaml` keeps Kubernetes manifest rendering in the
  existing CI flow with `./gradlew k8sRenderDev --no-daemon`.
* Existing Trivy secret scanning, image vulnerability scanning, OpenAPI drift
  checks, and Prometheus rule validation remain separate from Codacy.
* `README.md` and `README.ja.md` include Codacy grade and coverage badges.

## Pros and Cons of the Options

### Codacy with JaCoCo Coverage Upload and New-Change-First Gates

* Good, because it fills the repository's coverage and maintainability
  visibility gap.
* Good, because it integrates with GitHub pull requests without replacing local
  Gradle checks.
* Good, because new-change-first gates keep the rollout incremental.
* Good, because repository-scoped coverage upload can be guarded without
  elevating `GITHUB_TOKEN` permissions.
* Good, because as of this decision Codacy's Team plan is documented as free
  forever for open-source projects and includes GitHub integration,
  cloud-hosted scans, merge gates, trends, coverage reports, and merge policies.
* Bad, because the Codacy repository, status checks, quality gates, token, and
  badges require external setup outside the codebase.
* Bad, because false positives and threshold tuning require baseline triage.

### SonarQube Cloud

* Good, because it can provide code quality, coverage, pull request analysis,
  and new-code quality gate signals comparable to the required Codacy role.
* Good, because its Team plan supports Java, Kotlin, Kubernetes, YAML, and
  GitHub Actions analysis.
* Bad, because as of this decision its self-service free tier is limited to
  exploration with private projects up to 50k LOC, while the Team plan starts
  as a paid plan.
* Bad, because the plan fit is worse for this open-source repository's initial
  quality visibility layer than Codacy's documented open-source Team plan.

### Existing CI Only

* Good, because it keeps CI simple and fully repository-local.
* Good, because current checks already cover formatting, compilation, tests,
  secret scanning, image vulnerabilities, Kubernetes manifest rendering,
  OpenAPI drift, and Prometheus rules.
* Bad, because coverage trends, duplication, complexity, maintainability, and
  code-smell feedback remain mostly invisible.
* Bad, because pull requests do not get an independent quality signal focused on
  new debt.

### Replace Existing CI Checks with Codacy

* Good, because it would centralize more quality feedback in one service.
* Bad, because it would weaken focused gates that are already well matched to
  this repository, such as Spotless, Error Prone, Gradle tests, Trivy,
  `k8sRenderDev`, and `promtool`.
* Bad, because Codacy is not the intended authority for the repository's
  functional tests, OpenAPI snapshot drift, Kubernetes rendering, or image
  vulnerability gate.
